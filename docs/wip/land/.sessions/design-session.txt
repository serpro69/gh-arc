  Section 3: Workflow Architecture

  Following the same pattern as diff, land gets its own internal/land/ package
  with a workflow orchestrator and focused sub-components.

  Package structure:
  - internal/land/workflow.go ‚ÄĒ LandWorkflow orchestrator, main Execute() entry
  point
  - internal/land/checks.go ‚ÄĒ pre-merge verification (approval, CI, dirty
  working dir)
  - internal/land/merge.go ‚ÄĒ merge execution (commit message formatting, API
  call)
  - internal/land/cleanup.go ‚ÄĒ post-merge cleanup (switch to trunk, pull, delete
   branch)
  - internal/land/output.go ‚ÄĒ terminal output formatting (progress steps, status
   indicators)
  - cmd/land.go ‚ÄĒ thin Cobra command, flag parsing, wires up workflow

  Core types:

  LandWorkflow holds the same dependency shape as DiffWorkflow: *git.Repository,
   *github.Client, *config.Config, owner/name strings. Sub-components are
  created in the constructor.

  LandOptions carries flag values: Squash, Rebase, Force, Edit, NoDelete bools.

  LandResult captures what happened: the merged PR, merge method used, commit
  SHA, branches deleted, messages collected for output.

  Workflow sequence (inside Execute()):
  1. Check working directory is clean ‚Üí fail if dirty
  2. Get current branch ‚Üí fail if on trunk
  3. Find PR for current branch ‚Üí fail if none
  4. Enrich PR with reviews + checks (parallel API calls, already exists in
  github.Client)
  5. Run pre-merge checks (approval, CI) ‚Üí fail/prompt/force based on config +
  flags
  6. Prepare commit message (PR title + body, optionally open $EDITOR)
  7. Call GitHub merge API
  8. Checkout default branch + pull latest
  9. Delete local branch (unless --no-delete)
  10. Print summary with restore command

  The cmd/land.go file follows the same thin pattern as cmd/diff.go: load
  config, open repo, create client, create workflow, execute, format output.

‚óŹ Section 4: Pre-merge Checks

  The checks module (internal/land/checks.go) runs verification before allowing
  the merge. Checks are evaluated in order, failing fast on the first blocking
  issue (unless --force).

  Check 1: Working directory clean
  - Uses existing repo.GetWorkingDirectoryStatus()
  - If dirty: always blocks, even with --force. This is a local safety invariant
   ‚ÄĒ merging with uncommitted changes would cause problems when switching
  branches. Not configurable.

  Check 2: Not on trunk
  - If current branch is the default branch (main/master): hard fail. "You're on
   main ‚ÄĒ nothing to land."
  - Not bypassable with --force (nonsensical to land trunk onto itself).

  Check 3: PR exists
  - Uses existing client.FindExistingPRForCurrentBranch()
  - If no open PR: fail with "No open PR found for branch 'feature/xyz'. Run gh 
  arc diff first."
  - Not bypassable with --force.

  Check 4: Approval status
  - Enrich PR with reviews via client.EnrichPullRequest()
  - Use existing DeterminePRStatus() to get aggregate review status
  - Behavior depends on requireApproval config:
    - "strict" (default): blocks if not approved. --force bypasses.
    - "prompt": warns and asks confirmation. --force bypasses the prompt.
    - "none": skips check entirely.
  - Error messages are actionable: "PR needs 1 more approval", "PR has
  outstanding change requests from @bob"

  Check 5: CI status
  - Behavior depends on requireCI config:
    - "required": query branch protection rules to get required checks list,
  verify those pass. --force bypasses.
    - "all": all check runs must be success/skipped/neutral. --force bypasses.
    - "none": skips check entirely.
  - In-progress checks: treated as failure (blocking). The user should wait or
  --force.
  - Error messages list which checks failed: "CI check 'tests' is failing", "CI
  check 'lint' is in progress"

  Check 6: Dependent PRs (informational)
  - Uses existing client.FindDependentPRs()
  - If found: warn but proceed. "Warning: 2 PRs target this branch ‚ÄĒ they will
  be retargeted after merge."
  - Never blocks.

‚óŹ Section 5: Merge Execution

  The merge module (internal/land/merge.go) handles commit message preparation
  and the GitHub merge API call.

  Commit message preparation:

  Default (no --edit):
  - Title: PR title as-is
  - Body: PR body as-is (includes any stacking metadata, test plan, etc.)

  With --edit:
  - Write a temp file with PR title on line 1, blank line, PR body below
  - Open $EDITOR (reuse the same editor detection logic from internal/template/)
  - Parse result: first line = title, rest = body
  - If user exits without saving or empties the file: abort merge, no harm done

  GitHub merge API call:

  Uses PUT /repos/{owner}/{repo}/pulls/{number}/merge with:
  - merge_method: "squash" or "rebase" (from flag or config)
  - commit_title: prepared title (for squash ‚ÄĒ rebase ignores this)
  - commit_message: prepared body (for squash ‚ÄĒ rebase ignores this)

  This is a new method on github.Client ‚ÄĒ something like MergePullRequest(ctx, 
  owner, repo, number, *MergeOptions). Returns the merge commit SHA on success.

  Error handling:
  - 405 Method Not Allowed ‚Üí merge method not allowed by repo settings.
  Actionable: "Squash merges are not allowed on this repo. Try --rebase or
  update repo settings."
  - 409 Conflict ‚Üí merge conflicts exist. "PR has merge conflicts. Resolve
  conflicts and push before landing."
  - 422 Unprocessable ‚Üí PR not mergeable (e.g., required checks haven't passed
  server-side). "GitHub blocked the merge ‚ÄĒ check branch protection rules."

  Each error maps to a specific, actionable message rather than a raw API dump.

  Does this look right?

‚óŹ Section 6: Post-merge Cleanup

  The cleanup module (internal/land/cleanup.go) runs after a successful merge.
  Each step is sequential since they depend on each other.

  Step 1: Checkout default branch
  - Get default branch name from config (github.defaultBranch) or detect via
  repo.GetDefaultBranch()
  - Run git checkout <default-branch>
  - If checkout fails: non-fatal. Print warning with manual command, skip
  remaining steps. The merge already succeeded ‚ÄĒ cleanup failure shouldn't look
  like a merge failure.

  Step 2: Pull latest
  - Run git pull origin <default-branch> to incorporate the just-merged commit
  - If pull fails: non-fatal warning. The user can pull manually.

  Step 3: Delete local branch
  - Skip if --no-delete flag or deleteLocalBranch: false in config
  - Capture the branch tip SHA before deleting (for the restore message)
  - Run git branch -D <branch-name> (-D because the branch is merged remotely
  but the local tracking info may not reflect that yet, so -d could refuse)
  - On success, print: Deleted local branch feature/auth (use git checkout -b 
  feature/auth <sha> to restore)
  - If delete fails: non-fatal warning. Branch already merged, user can clean up
   manually.

  Design principle: The merge is the critical operation. Everything after it is
  best-effort cleanup. If any cleanup step fails, we report it clearly but never
   return an error exit code ‚ÄĒ the merge succeeded and that's what matters. The
  output makes it obvious what happened and what didn't, with manual recovery
  commands for anything that failed.
